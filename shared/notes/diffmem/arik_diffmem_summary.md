# DiffMem Summary - Arik's Notes

## 1. Overview and Core Principles

DiffMem is a lightweight, Git-based memory backend for AI agents. It leverages Git's version control capabilities to store and track the evolution of memories in Markdown files.

**Key Concepts:**
*   **Git-Based**: Uses Git for versioning, diffs, blame, and history tracking.
*   **Markdown Files**: Memories are stored in human-readable Markdown files.
*   **Current-State Focus**: Files present the "now" view of information, reducing context overhead for LLMs. Historical data is in Git history.
*   **Differential Intelligence**: Git diffs allow agents to easily track how memories evolve over time.
*   **Durability and Portability**: Plaintext Markdown and Git's distributed nature ensure long-term usability and backup.
*   **Efficiency for Agents**: Separates "surface" (current files) from "depth" (Git history) for lean context windows.
*   **Brain-Inspired**: Simulates hierarchical organization, reconstructive retrieval, and adaptive strengthening.

## 2. Architecture: Orphan Branches & Worktrees

DiffMem uses a Federated Git Architecture for user isolation and scalable storage.

*   **Storage Repository**: A single, hidden Git repository (`/home/zyra/.hermes/DiffMem/`) holds all history.
*   **Orphan Branches**: Each user gets an isolated `orphan branch` (e.g., `user/alex`) within the storage repository. These branches share no history.
*   **Git Worktrees**: When a user is active, their branch is checked out into a `Git Worktree` (e.g., `/app/active_contexts/{user_id}/`). Agents operate within this worktree, which appears and functions as a standard Git repo root.

**Repository Structure (Per User Worktree):**
```
<worktree_root>/
├── repo_guide.md       # DiffMem's guide for memory structure
├── index.md            # Autogenerated: keyword lookup, semantic index summary
├── memories/           # Core memory storage
│   ├── people/         # Biographical: profiles (e.g., 'alex.md')
│   ├── contexts/       # Factual/semantic: themes (e.g., 'health.md')
│   └── episodes_index.md # Autogenerated: thematized groupings of episodes
├── timeline/           # Episodic: chronological
│   └── YYYY-MM.md      # Monthly timeline files (e.g., '2024-05.md')
└── .git                # Git internal files (pointing to central storage)
```

## 3. Writer Agent

The `WriterAgent` is responsible for ingesting session transcripts and updating the memory repository.

**Process (`process_session`):**
1.  **Identify Entities**: Uses LLM to find entities to create or update in the session.
2.  **Create New Entities**: LLM generates content for new entities, creating `Markdown` files (e.g., `memories/people/new_person.md`).
3.  **Update Existing Entities**: LLM generates `replace`, `insert_after`, or `append` operations for existing files (e.g., entity profiles, context files). These updates are applied.
4.  **Create Timeline Entry**: Generates a timeline entry based on Git diffs and session input, appending to `timeline/YYYY-MM.md`.
5.  **Build Entity Indexes**: For modified files, an LLM generates a semantic index (JSON metadata) which is embedded within the Markdown file under a `## SEMANTIC INDEX` section.
6.  **Rebuild Master Index**: Compiles `index.md` from all entity semantic indexes, sorted by "memory strength".
7.  **Commit Session**: Stages all changes and commits them to Git.

**Key Features:**
*   **LLM-Driven**: Heavily relies on LLM calls (via OpenRouter) for content generation and update logic.
*   **Parallel Processing**: Uses `ThreadPoolExecutor` for concurrent LLM calls when creating/updating entities.
*   **Memory Strength**: Calculates a strength score for memories based on edit frequency and recency.

## 4. Retrieval Agent

The `RetrievalAgent` is a multi-turn LLM agent designed to fetch relevant context from the memory repository.

**Process (`run_retrieval_agent`):**
1.  **System Prompt**: Injected with `user_id`, `max_tokens`, and `baseline_tokens`.
2.  **User Message**: Takes the conversation history as input.
3.  **LLM Interaction**: Iteratively calls an LLM (OpenAI-compatible API) with a single `run(command="...")` tool.
4.  **Command Execution**: The LLM uses commands like `cat`, `grep`, `git log`, `git diff`, `git blame`, `ls`, `wc` to explore the git repository.
5.  **Retrieval Plan**: The LLM outputs a `RetrievalPlan` (JSON) containing pointers to relevant files/sections and a synthesis.

**Key Features:**
*   **Git-Native Exploration**: Directly uses Git commands via a sandboxed shell.
*   **Deterministic Baseline**: `baseline.py` loads essential context like user entity, recent timeline, and `[ALWAYS_LOAD]` blocks, ensuring core information is always available regardless of retrieval agent's LLM choices.
*   **Token Efficiency**: Only loads necessary context by exploring the repository intelligently.

## 5. API Endpoints

DiffMem exposes a FastAPI service with key endpoints for memory management:

*   `POST /memory/{user_id}/onboard`: Creates a new user.
*   `POST /memory/{user_id}/process-and-commit`: Ingests a session transcript and commits changes.
*   `POST /memory/{user_id}/context`: Retrieves context for a conversation.

## 6. Integration Potential with Triple-Hit Memory

DiffMem can significantly enhance our existing PostgreSQL, Neo4j, and local files memory system:

*   **Versioned Local Files**: DiffMem provides a robust, version-controlled implementation for our "local files" memory tier. Git's history acts as a complete audit log.
*   **Enriched PostgreSQL Embeddings**: The semantic indexes generated by DiffMem's `WriterAgent` (embedded JSON in Markdown) can provide richer, more structured text for generating vector embeddings in PostgreSQL. This could lead to more precise semantic search.
*   **Dynamic Neo4j Graph**: The `WriterAgent` identifies and updates entities. This information, along with the relationships implied by Markdown links and the Git commit history (showing how facts evolve), can be directly fed into Neo4j to build a more dynamic and detailed knowledge graph. The "memory strength" could also influence graph node weights.
*   **LLM-Guided Data Ingestion/Retrieval**: DiffMem's `WriterAgent` and `RetrievalAgent` provide LLM-guided mechanisms for both writing and reading memories, which can be adapted or integrated into our existing memory operations.
*   **Unified Context Management**: The `RetrievalPlan` from DiffMem could inform which context pieces (from PostgreSQL, Neo4j, or directly from DiffMem's worktree) are most relevant for a given query, creating a more cohesive context for our main LLM.
*   **Cron Session Enhancement**: The curl commands for DiffMem (as noted in `GITHUB_ACCESS.md` and my memory) can be directly integrated into cron jobs for autonomous memory updates and retrievals, bridging the gap for cron sessions where the `memory` tool is not available.
